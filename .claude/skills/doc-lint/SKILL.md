# doc-lint — 문서 드리프트 탐지

문서가 실상태보다 낡으면 다음 세션이 낡은 전제로 작업한다. 이 스킬은 **모순 탐지·보고 전용**이다.
수정은 별도 이슈/PR로 — 이 스킬 실행 중 문서를 편집하지 않는다.

## 1. 실상태(ground truth) 수집

```bash
gh issue list --state open --limit 100
gh issue list --state closed --limit 40
gh pr list --state merged --limit 30
gh issue view 9            # 로드맵 에픽 — 진행률·체크박스 상태
git log --oneline -30 origin/main
```

## 2. 대상 문서와 점검 축

대상: 레포 루트 `CLAUDE.md`, `README.md`, 로드맵 에픽 이슈 **#9** 본문(진행 상태·체크박스). Docs 디렉터리가
생기면 `Docs/*.md`도 포함(단, 파일명에 날짜가 박힌 감사 스냅샷은 불변 원본이므로 제외).

각 문서에서 다음을 추출해 실상태와 대조:

1. **이슈/PR 참조(`#N`)의 상태 주장** — "미구현", "예정", "진행 중", "완료(진행 중)"로 표기된 #N이 실제로는
   닫혔는지. 반대로 "완료"로 표기됐는데 열려 있는지. (예: #9의 진행률 `0/8`, "다음 작업 → #1"이 실제와 맞는지.)
2. **열거 목록** — 로드맵 단계 목록, 명령어 표, 라벨 체계 등. 항목별 실상태 확인. **열거는 드리프트의 주원인** —
   낡은 열거는 "갱신"보다 "포인터(gh 명령 한 줄)로 대체"를 우선 제안한다.
3. **날짜 표기**("기준", "착수") — 그 날짜 이후 머지된 PR 중 해당 주장을 바꾸는 것이 있는지.
4. **명령어·경로 실재** — 문서의 `npm run <script>`가 `package.json`에 있는지, 참조 파일 경로가 존재하는지.
5. **수치 주장**(테스트 개수 등) — 검증이 싸면 확인, 아니면 "썩는 수치, 제거 권고"로 표시.
6. **문서 간 모순** — 같은 사실을 CLAUDE.md와 README가 다르게 말하는 경우.

## 3. 보고 형식

모순만 보고한다(정상 항목 나열 금지). 심각도 순:

- **P1 — 오도**: 다음 세션이 잘못된 작업을 하게 만드는 것 (완료된 걸 미구현으로, 삭제된 명령어 안내)
- **P2 — 낡음**: 사실과 다르지만 작업 방향은 안 바꾸는 것 (날짜, 수치)
- **P3 — 구조 권고**: 썩기 쉬운 표면 자체의 제거 제안 (열거 → 포인터 대체)

각 항목: `파일:줄`(또는 `이슈 #N`) · 문서의 주장 · 실상태 · 근거(이슈/PR 번호). 마지막에 수정 PR로 묶을 제안 요약.

## 4. 주의

- 문서 수정이 필요하면 이 스킬 실행 중이 아니라 별도 브랜치/PR로(CLAUDE.md 워크플로 규칙: main 직접 커밋 금지).
- 메모리 디렉터리(`~/.claude/projects/...`)는 이 스킬 범위 밖.
